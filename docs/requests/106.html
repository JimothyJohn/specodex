SCHEMA Phase 4 — kg → kgf → N Force coercion

Lintech (and other industrial vendors) publishes load capacities like {value: 703, unit: 'kg'} meaning kilogram-force, not mass. The Force family already handles kgf → N at ×9.80665 — what was missing was the catalog-convention rewrite for bare kg. This PR adds a per-family pre_rewrite table on UnitFamily, wires kg → kgf for FORCE only, and logs at WARNING so the coercion is auditable.

▮▮▮ WHAT CHANGES ▮▮▮
  1. A-1

    UnitFamily gains a pre_rewrite map

    New optional dict[str, str] field on the frozen dataclass, plus a rewrite() method that returns the rewritten unit (or the input unchanged). Family-scoped — Mass-typed fields still treat kg as canonical mass. Wrong-family kg on a Voltage field still drops to None as before.

  2. A-2

    FORCE family declares {"kg": "kgf"}

    Both _typed_value_unit and _typed_min_max_unit apply the rewrite to incoming dict / ValueUnit / MinMaxUnit inputs before the family check, so the existing kgf→N (×9.80665) normaliser handles the conversion in one pass. logger.warning("force field: rewriting unit 'kg' to 'kgf' (catalog convention)") fires once per coercion so operators can grep for catalog-convention coercions when auditing extraction quality.

  3. A-3

    9 new test cases under TestForceKgCoercion

    Covers: dict input, ValueUnit instance input, MinMaxUnit + scalar collapse, ForceRange (range- typed), Mass-field-still-canonical, Voltage- field-still-rejects-kg, WARNING-log emitted on rewrite, no-warning on plain kgf.
