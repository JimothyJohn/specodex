  1. E-1

    Header — always-on

    New Feedback button in the header-options group, sitting between the OPTIONS label and the Density / Theme / Account toggles. Opens the modal with no pre-fill — the General path for users who came in to send something specific without a context to anchor it.

  2. E-2

    Empty state — "I filtered to zero"

    Replaces the flat "No results match your specs" at ProductList.tsx:1020 with a follow-up card. Pre-selects "I can't find what I need" and threads the active product type plus serialized filter set into the mail body. The highest-signal moment — the user just demonstrated a coverage or filtering gap.

  3. E-3

    Empty state — "DB is empty for this type"

    Same card, different copy. When products.length === 0 the button reads "Tell us what to add" and pre-selects "A product is missing". Hits early-stage product types where the catalog has the schema but no rows yet.

▮▮▮ MAIL BODY SHAPE ▮▮▮
Subject: [Specodex] I can't find what I need (motor)

<user message>

---
Sent from the Specodex feedback form.
Category: I can't find what I need
Route: /?type=motor
Product type: motor
Active filters:
  - Rated Voltage >= 200
  - NOT Manufacturer = Acme

Triage-friendly: each filter on its own line with the human-readable displayName instead of the dot-notation key, and NOT-prefix on mode: 'exclude' so the inversion isn't lost. Neutral / empty filters are dropped.

▮▮▮ WHY MAILTO ▮▮▮

Zero infra, no auth, no PII to design, no rate-limit / abuse surface. buildFeedbackMailto() is a pure function returning a mailto: URL; submit assigns it to window.location.href so the user's mail client opens with the body already filled. They edit before sending if they want; if they don't, what arrives reads like a small bug report.

A POST /api/feedback Lambda → DynamoDB or SES is the obvious follow-up if volume warrants. That pulls in app/infrastructure/** which is off-limits for autonomous work — it needs explicit sign-off and an IAM policy rev. Not a regression risk to ship the mailto path now and graduate later.

▮▮▮ APP-NATIVE CHROME ▮▮▮
  1. S-1

    Modal scaffolding

    Reuses .confirm-dialog-overlay / .confirm-dialog — same overlay, fade animation, focus return, Esc-to-cancel as the existing ConfirmDialog. No <dialog>, no UA backdrop.

  2. S-2

    Form validation

    noValidate on the form per todo/STYLE.md; UA validation bubbles are banned. Empty message is allowed (helper inserts a (No additional message) placeholder) — submission with just a category + context is still useful signal.

  3. S-3

    No alert / confirm

    Submit dispatches the mailto in the same tab and closes the modal. No success toast yet — the mail-client launch is the visible feedback. If we add a success toast later it goes through useToast(), not window.alert.

▮▮▮ WHAT THIS PR DOESN'T DO ▮▮▮
  1. N-1

    No backend endpoint

    No POST /api/feedback, no DynamoDB table, no SES wiring. Mailto is the entire submission path. Graduating to a backend is a follow-up gated on actual volume.

  2. N-2

    No per-product report link

    The proposal included a "Report this listing" link inside ProductDetailModal for spec-level corrections. Trimmed from V1 — global modal handles "wrong spec" feedback today; per-row reporting can land in a follow-up if the global category proves too coarse for triage.

  3. N-3

    No proactive nudges

    The user comes to the modal; the modal doesn't come to the user. Toast prompts after N filtered-to-zero events, post-search surveys, etc. are deliberately out of scope — discoverable but not pushy is the V1 stance.
