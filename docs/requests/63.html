STYLE Phase 2 — ConfirmDialog primitive replaces window.confirm

window.confirm blocks the JS thread, renders in OS chrome that can't be themed, and traps tests in a synchronous dialog the runner can't dismiss. The new ConfirmDialog primitive is async, themed, keyboard-friendly, and lets tests resolve programmatically by clicking the rendered buttons.

▮▮▮ THE NEW API ▮▮▮

Imperative, mirroring window.confirm — no JSX wrapper, no open/setOpen ceremony at the call site: 

const confirm = useConfirm();
if (!(await confirm({
    title: 'Delete project?',
    body: '"Foo" and all its product references will be removed.',
    confirmLabel: 'Delete',
    confirmVariant: 'danger',
}))) return;
▮▮▮ WHAT'S NEW ▮▮▮
  1. API

    Provider + hook

    app/frontend/src/components/ui/ConfirmDialog.tsx exports ConfirmProvider + useConfirm(). Provider holds at most one pending confirm; a second call cancels the first (resolves false) and replaces it.

  2. UX

    Keyboard + focus

    Esc / backdrop click / Cancel resolve false; Enter / Confirm resolve true. Returns focus to the trigger element on close. 120 ms fade-in respects prefers-reduced-motion.

  3. STYLE

    Themed via tokens

    Uses existing --bg-primary / --border / --accent-primary. Danger variant uses #b03232 (matches AuthModal). No new design tokens introduced.

  4. WIRING

    Mounted in App.tsx

    <ConfirmProvider> sits between AppProvider and BrowserRouter so the whole route tree can call useConfirm().
