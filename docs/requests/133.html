Pin tsconfig types — unblock cdk bootstrap on TS 6.

PR #132 just unblocked the backend Lambda bundle step. The next step in the deploy chain — cdk bootstrap — has been failing the same way since 2026-05-03 with Cannot find name 'path' and Cannot find name '__dirname'. Root cause: Dependabot PR #22 bumped typescript 5.9.3 → 6.0.3 in /app/infrastructure, and TS 6 changed typeRoots auto-discovery — with no explicit types field, @types/node is silently dropped. One-line fix.

▮▮▮ WHAT WAS BROKEN ▮▮▮
  • cdk bootstrap died at compile time. cdk.json declares "app": "npx ts-node bin/app.ts". ts-node compiles the entry file with infra's tsconfig.json. After 2026-05-03 every compile reported lib/api-stack.ts(10,23): TS2591 Cannot find name 'path' and lib/api-stack.ts(37,45): TS2304 Cannot find name '__dirname'. Bootstrap never reached the CDK app.
  • TS 6 changed typeRoots auto-discovery. The infra tsconfig declares "typeRoots": ["./node_modules/@types", "../node_modules/@types"] and no types field. TS 5 honored an implicit-include rule (every @types/* package under typeRoots auto-loaded). TS 6's getAutomaticTypeDirectiveNames returns [] in this config — verified locally: node -e "console.log(ts.getAutomaticTypeDirectiveNames({typeRoots: […]}, host))" → [].
  • Result: @types/node never loaded. import * as path from 'path' and the global __dirname were both undefined at compile time. Locally app/infrastructure/node_modules/@types/node/ still exists at 25.6.0 — the file is right there, TS 6 just refuses to look at it.
  • Why now. Dependabot PR #22 merged 2026-05-03. Every master push between then and 2026-05-10 either cancelled before the deploy step ran or was caught by PR #132's lockfile bug. PR #132 just merged, the bundle step now succeeds, this is the next layer of the same onion. Prod hasn't shipped via CI since 2026-05-02.
▮▮▮ THE FIX ▮▮▮

Declare "types": ["node", "jest"] explicitly. node covers everything under lib/*.ts (every file that imports 'path' or references __dirname). jest covers test/*.test.ts (auto-included previously via @types/jest sitting in node_modules/@types; without it, tsc --noEmit in the build script fails 33 times in test/waf-alarms.test.ts).

The list is closed: @types/node and @types/jest are the only @types/* packages in the infra dep tree. Pinning them explicitly matches the previous implicit-include behavior exactly — no expansion, no surprise. typeRoots stays in place so that if a future dep adds a third @types/* package, we can extend the list without re-deriving the resolution paths.
