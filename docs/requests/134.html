▮▮▮ PULL REQUEST #134 ▮▮▮

Skip cdk bootstrap when the toolkit is current — third layer of the same onion.

PR #132 fixed the backend bundle lockfile. PR #133 fixed the TS6 tsconfig. With those gone, the same deploy run hits a third failure: npx cdk bootstrap tries to clean up a no-op change set on CDKToolkit and the CI role lacks cloudformation:DeleteChangeSet. Fix: skip bootstrap when the live stack is healthy and on a recent enough version. No IAM mutation.

▮▮▮ WHAT WAS BROKEN ▮▮▮
  • Bootstrap ran on every deploy. cli/quickstart.py's cmd_deploy called npx cdk bootstrap aws://<account>/<region> unconditionally. The script's been that way since the AWS deploy step landed (commit 09500c8) on the theory that cdk bootstrap is idempotent.
  • Idempotent ≠ permissionless. The cdk@2.1120.0 bootstrap CLI still creates a no-op change set against CDKToolkit, computes the diff, and deletes the change set. That round-trip requires cloudformation:{Create,Describe,Execute,Delete}ChangeSet. The gh-deploy-datasheetminer OIDC role grants DescribeStacks / DescribeStackEvents / GetTemplate via the ReadCloudFormationStateForCdkCli Sid, but no change-set actions.
  • Result. Every Deploy Staging after PR #133 merged dies with AccessDenied: User: arn:aws:sts::***:assumed-role/gh-deploy-datasheetminer/GitHubActions is not authorized to perform: cloudformation:DeleteChangeSet on resource: stack/CDKToolkit/<uuid>. Same chain as #132 and #133 — the failure moved one step downstream each time the prior layer landed.
  • Why now. Bootstrap has needed DeleteChangeSet for ages — the cdk CLI behavior here predates the project. Earlier deploys had it because the IAM policy used to include it (or used a broader-scoped CFN allow that masked the gap). At some point between then and 2026-05 the policy was scoped down and the gap surfaced the next time the deploy actually completed past the bundle and synth steps. Pinning the exact policy commit is a follow-up — not blocking the fix.
▮▮▮ THE FIX ▮▮▮

Add cdk_toolkit_is_current(region) to cli/quickstart.py. It runs aws cloudformation describe-stacks --stack-name CDKToolkit, parses the result, and returns True iff the stack is *_COMPLETE (not in any ROLLBACK state) AND its BootstrapVersion output is ≥ a constant _CDK_BOOTSTRAP_MIN_VERSION = 21. The deploy code wraps the bootstrap call in if cdk_toolkit_is_current(region): info("…skipping bootstrap").

Why 21: that's the asset-publishing-role era. Every cdk-lib release in the last two years has worked against a v21+ bootstrap. The live stack is at v30 (last updated 2026-03-21). If a future cdk-lib needs newer, cdk deploy itself fails with a clear "bootstrap version too old" message — we re-bootstrap once from a local terminal with permissions, and CI resumes skipping. The version floor stays as a single constant near the helper so the next bump is one-line.

The DescribeStacks call is free. The OIDC role's CdkDeploy policy already grants cloudformation:DescribeStacks via the ReadCloudFormationStateForCdkCli Sid (added when ./Quickstart cdk-outputs shipped). No IAM mutation needed for this PR. Alternative considered: expand the role to add the change-set actions on stack/CDKToolkit/*. Rejected here in favor of the reversible in-repo fix; can revisit if bootstrap upgrades start landing more than once a year.
