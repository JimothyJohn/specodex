▮▮▮ WHAT GETS EMITTED ▮▮▮

Two new exports appear at the bottom of generated.ts: 

export const PRODUCT_TYPES = [
    'contactor', 'drive', 'electric_cylinder',
    'gearhead', 'linear_actuator', 'motor', 'robot_arm',
] as const;
export type ProductTypeLiteral = (typeof PRODUCT_TYPES)[number];
export type Product = Contactor | Drive | ElectricCylinder
                      | Gearhead | LinearActuator | Motor | RobotArm;
▮▮▮ WHAT CHANGES ▮▮▮
  1. GEN

    scripts/gen_types.py

    New _product_union_type() walks SCHEMA_CHOICES.values(), sorts by class name, and emits export type Product = X | Y | Z;. main() appends both the existing _product_types_constant() and the new union to the pydantic2ts output.

  2. TS

    generated.ts regenerated

    The actual Product + ProductTypeLiteral + PRODUCT_TYPES lines at the bottom — what the codegen now produces.

  3. DOC

    MODELGEN.md updated

    OQ2 marked ✅ resolved with the implementation reference; suggested-order step 5 updated to note the union is now generated.

▮▮▮ WHY IT'S SAFE TO SHIP INDEPENDENTLY ▮▮▮
  1. S-1

    Purely additive

    No consumer currently imports Product from generated.ts (they import the hand-typed one from models.ts), so this can't break compilation.

  2. S-2

    Phase 0a-ii unblocked

    The rewired models.ts shim (PR #60) can now export type { Product, ProductTypeLiteral } from './generated'; instead of hand-typing.

  3. S-3

    Auto-widens

    Adding a new product type under specodex/models/ auto-widens the Product union — one less hand-edit in the runbook.
