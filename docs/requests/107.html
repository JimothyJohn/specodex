HARDENING Phase 3.3 — schema forward/backward compat tests

tests/unit/test_models_roundtrip.py covers same-version roundtrip (construct, dump, reload). It catches a wide class of bugs but not the cross-version ones — a deploy that adds a required field breaks existing rows; a deploy that adds a field then gets rolled back breaks the rollback. This PR adds frozen JSON snapshots per ProductType with deterministic UUIDs, plus a compat-test pair that runs them in both directions.

▮▮▮ WHAT CHANGES ▮▮▮
  1. A-1

    Frozen JSON snapshots, one per ProductType

    tests/unit/fixtures/schema_snapshots/<type>.json for each of the 7 product types (contactor, drive, electric_cylinder, gearhead, linear_actuator, motor, robot_arm). Synthetic per CLAUDE.md "Never seed dev or test from prod data — synthetic only." Deterministic UUID5 product_ids from a fixed namespace so regens produce zero diff on a clean tree.

  2. A-2

    Forward + backward compat directions

    TestForwardCompat: every snapshot loads under current Pydantic, and the load → dump → load round-trip is pure. TestBackwardCompat: extra unknown fields injected into the snapshot are ignored by validation (Pydantic's extra="ignore" default); stripping every nullable field still validates (no field has silently become required).

  3. A-3

    Drift gate with refresh helper

    TestSnapshotsExist fails if a new ProductType is added without a snapshot — the assertion message tells the contributor exactly how to refresh (refresh_snapshots() exposed from the test module). CLAUDE.md "Adding a new product type" gains step 5 wiring it up.
