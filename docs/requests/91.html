▮▮▮ PULL REQUEST #90 ▮▮▮

DOUBLE_TAP — structured encoder schema + verifier-loop extraction

The encoder_feedback_support field on Motor / Drive / ElectricCylinder / LinearActuator was a str / List[str] soup — vendors say "EnDat 2.2", "BiSS-C 26-bit", "Smart Absolute", and the LLM emits whatever string it sees. Cross-product matching (drive ↔ motor compatibility) degenerates to string comparison and false-fails on casing or version drift. This PR closes the taxonomy (13 device types × 22 wire protocols, mirroring Heidenhain's published catalog axis) and adds a verifier-loop runner that re-prompts Gemini with the first-pass output + per-field hunting captions when anything looks ambiguous or missing — the double-tap.

▮▮▮ WHY ▮▮▮

Two failures stapled together because they share a root cause. (1) The schema is too loose. The compatibility layer (integration/compat.py:_compare_feedback) does string membership against vendor-flavored free-text — "EnDat 2.2" doesn't equal "endat 2.2" doesn't equal "EnDat-2.2". We mask the problem by _soften-ing every fail to partial, which hides bugs. (2) One-shot extraction has no recovery. When Gemini punts on a field — omits it, picks a wrong unit, or fills fuzzy text where structured belongs — we either trip the quality gate at min_quality=0.25 or, worse, the row sails through with ambiguous strings and we blame the vendor for an incomplete catalog the catalog actually publishes.

▮▮▮ WHAT CHANGES ▮▮▮
  1. A-1

    EncoderFeedback model + closed taxonomy

    New specodex/models/encoder.py defines EncoderDevice (13 values: incremental_optical, absolute_optical_multiturn, resolver, none, unknown, …) and EncoderProtocol (22 values: biss_c, endat_2_2, hiperface_dsl, mitsubishi_j5, yaskawa_sigma, drive_cliq, oct_beckhoff, …). Both axes carry an "unknown" sentinel + a raw passthrough field so the verifier has something to grab when the LLM punts. Disambiguation rules from the taxonomy doc are encoded in the field docstrings (which Gemini sees in response_schema): bare "EnDat" → endat_2_2; "Hiperface" without "DSL" → classic hiperface; "N-bit absolute" with no vendor → bits_per_turn=N + protocol=null (do NOT guess).

  2. A-2

    Back-compat: legacy free-text rows still validate

    EncoderFeedback._coerce_legacy_freetext is a model_validator(mode="before") that runs string inputs through synonym tables — "EnDat 2.2 multi-turn 26-bit" becomes {protocol: endat_2_2, device: absolute_optical, mode: absolute, multiturn: true, bits_per_turn: 26, raw: ...}. Drive's protocol list gets the same treatment via a BeforeValidator on the field. Old DB rows survive the upgrade without a migration; rows whose text the synonym tables can't fully resolve get device="unknown" + raw=<input> so the verifier flags them.

  3. B-1

    Compat layer goes typed

    _compare_feedback now extracts the motor's EncoderFeedback.protocol and checks membership in the drive's List[EncoderProtocol], with a SUBSUMES widening matrix (EnDat 2.2 accepts 2.1 per Heidenhain docs; Hiperface ≠ Hiperface DSL — distinct cable architectures, do not collapse). The motor side with no protocol set returns partial rather than fabricating a match.

  4. C-1

    Verifier — first-pass quality probe

    New specodex/double_tap/verifier.py inspects each parsed product and emits a Probe describing what to re-extract. v1 fires on (a) encoder ambiguity (device or protocol resolved to unknown) and (b) common-field missing (per-product-type small list of "almost always populated" fields: motor: rated_voltage, rated_torque, rotor_inertia, …). Each FieldProbe carries the catalog captions to hunt for — for rotor_inertia: "Rotor inertia", "GD²/4", "J", "kg·m²", "kg·cm²".

  5. C-2

    Runner — primed second pass + conservative merge

    specodex/double_tap/runner.py: first-pass extract → verify → if any probe fires, build ONE second-pass prompt covering every probed variant (so cost is one extra LLM call regardless of variant count, not N) → re- extract → merge. The merge takes second-pass values only for probed fields; first- pass values for everything else. A future contributor "improving" the merge to prefer second-pass everywhere would let a hallucinating second pass regress good first-pass data — the bench's regression-direction column catches that.

  6. D-1

    Scraper integration — env-gated, off by default

    SPECODEX_DOUBLE_TAP=1 flips the bundled-PDF and HTML extraction paths in process_datasheet to use the runner instead of call_llm_and_parse. Per-page extraction stays single-shot for cost. Telemetry (double_tap_fired, double_tap_probes_fired, double_tap_fields_recovered, double_tap_fields_corrected, double_tap_unresolved) writes to the ingest log so godmode can show daily probe-fire and recovery yield.

  7. E-1

    Bench A/B harness

    ./Quickstart bench --ab runs each fixture in BOTH modes and emits a comparison table: Δ recall, Δ precision, Δ tokens, Δ cost, and a "worth it?" verdict. Threshold encoded as WORTH_IT_THRESHOLD_PP_PER_100PCT_TOKENS = 5.0 — a fixture is worth the second pass when (Δ recall + Δ precision) per +100% tokens ≥ 5pp. The cache layout grows a <slug>.double_tap.json twin so live runs populate both modes once and offline --ab runs are free.

  8. F-1

    Frontend: typed encoder + auto-derived UI

    ./Quickstart gen-types regenerates generated.ts; the EncoderFeedback interface lands with full enum literals so the existing records-derived attribute logic (filters.ts:deriveAttributesFromRecords) auto-renders structured fields. Client-side compat.ts:compareDriveMotorFeedback updated to compare motor.encoder_feedback_support ?.protocol against the drive's protocol list (was passing the structured object to a Maybe<string> helper — caught by tsc).

▮▮▮ HOW TO VERIFY ▮▮▮
  1. V-1

    Pre-push gate

    ./Quickstart verify green: 1,202 pytest cases (53 new), 454 jest cases, 479 vitest cases, all type checks. Codegen drift check inside verify confirms the committed generated.ts matches what gen-types produces.

  2. V-2

    Bench A/B (live, behind operator gate)

    ./Quickstart bench --ab --live --update-cache on the existing fixture set populates both single-pass and double-tap caches and emits the comparison table. The PR un-drafts when at least 2 fixtures show worth_it=True (Δ recall + Δ precision ≥ 5pp per +100% tokens) — anything below that and the runner stays env-gated indefinitely. Fixture additions for encoder-specific stress (Mitsubishi MR-J5, Yaskawa Sigma-7) follow in a separate PR once baseline numbers are in.

  3. V-3

    Compat regression spot-check

    uv run pytest tests/unit/test_integration.py -v — all 20 cases pass. The "EnDat 2.2 motor on drive supporting [EnDat 2.2, Resolver]" test confirms the typed compat path returns ok. The "motor with EnDat 2.2 on drive supporting only [Resolver]" test confirms it returns fail (was sometimes false-passing under the old string-membership check).

▮▮▮ FOLLOW-UPS ▮▮▮
  1. N-1

    Delete _soften when API layer ready

    compat.py:_soften exists because the old string-membership check would false-fail on casing/version drift; softening every fail to partial hid the bug. The typed check makes softening unnecessary — but the API layer still calls _soften by default. Separate PR rips it out once a couple weeks of production data show no surprise fails.

  2. N-2

    Wrong-unit-dropped verifier rule

    Rule 3 from the plan (common.py's BeforeValidator silently nulling wrong-unit-family inputs) is stubbed. Needs the parser to surface the pre-coercion raw payload alongside the validated model. One-day add once the runner has bench numbers.

  3. N-3

    Generalize beyond encoders

    Drive.fieldbus, Motor.frame_size, Gearhead.ratio, IpRating — same vendor-string-soup problem, all benefit from the verifier framework once it's proven. Each is a one-day add after the bench shows positive ROI on encoder.
