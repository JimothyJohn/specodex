HARDENING Phase 3.4 — concurrent-write stress test for batch_create

scraper.batch_create and process_datasheet write shared DynamoDB state. No test exercised the codepath under concurrent writes with overlapping product_ids — the exact shape race conditions take in production when two scraper workers ingest the same datasheet. This PR adds a moto-backed stress test that fires 20 parallel writers against 5 contested IDs, with timing jitter to interleave writes across threads.

▮▮▮ WHAT CHANGES ▮▮▮
  1. A-1

    No exceptions under concurrency

    20 ThreadPoolExecutor writers issuing batch_create with overlapping IDs all return the expected success count. Catches any silent drop in the batch_writer codepath under contention.

  2. A-2

    No torn writes per ID

    Each writer signs every text field on its rows with the writer_id; the assertion phase reads each contested row back and verifies all text fields encode the SAME writer_id (not field-by- field mixing). DynamoDB's put_item is atomic at the item level — a failure means the serialiser split a logical write into multiple physical writes.

  3. A-3

    No rows lost or duplicated

    Distinct (product_type, product_id) keys after the stress run equal the contested-IDs count exactly. Last-write-wins semantics hold; no row escapes into a duplicate or vanishes.
