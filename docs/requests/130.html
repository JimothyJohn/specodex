▮▮▮ WHAT CHANGES ▮▮▮
  • New ? button per spec row. Lives at the right edge of every .spec-row and every nested .spec-subrow (so dimensions get one button per width / height / depth, not one for the whole group).
  • Click opens the existing FeedbackModal with defaultCategory="wrong_info". The heading swaps from “Send feedback” to “Report inaccurate value”, and the field label/value/unit show up in the mailto body as Field: … + Reported value: … lines.
  • Tooltip primitive, not native title=. Per STYLE.md Phase 1 — every action stays inside the app, no OS chrome. The button's hover label is themed and portaled like every other Tooltip.
  • Stacked-modal hygiene. The FeedbackModal mounts inside the parent's modalRef. The parent's outside-click and Escape handlers are gated while the sub-modal is open, so dismissing one no longer unwinds the other.
  • FeedbackModal.title → heading. The STYLE.md drift gate's blanket title= ban can't distinguish a custom React prop from the native HTML attribute. Renaming the prop dodges the gate naturally without an allowlist exception.
  • Per-component CSS file. ProductDetailModal.css follows the CLAUDE.md convention introduced for Tooltip / ConfirmDialog / FeedbackModal / Toast. App.css is unchanged — keeps it from being the universal merge target.
▮▮▮ WHY MAILTO ▮▮▮

The minimum viable shape of “flag a wrong spec” is a structured email — same pattern the existing feedback button already uses. Zero new infra, no auth / rate-limit / abuse surface to design, no DynamoDB schema, no IAM. Complaints land in the inbox where the future verifier pipeline can pick them up.

When volume warrants it, the natural follow-up is a POST /api/v1/complaints + DynamoDB row — utils/feedback.ts already flags that as the next move. Until then, a tighter feedback loop with a human in the loop beats a half-built pipeline.
