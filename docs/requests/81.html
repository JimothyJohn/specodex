▮▮▮ THE SAFELOG HELPER ▮▮▮
  1. H-1

    Why log-injection matters

    console.log(${req.path}) with no sanitization lets a caller send GET /foo%0A%0A[admin] deleted everything — the embedded \r\n renders as a real newline in CloudWatch, faking a separate log line. Auditors and incident response then chase a ghost. The fix is mechanical: strip control chars before interpolation.

  2. H-2

    What safeLog does

    New file app/backend/src/util/log.ts exports a single safeLog(value: unknown): string that: (1) coerces non-string inputs via JSON.stringify, (2) strips \r / \n / \t, (3) truncates at 200 chars with an ellipsis, (4) wraps the result in single quotes so reviewers can see at a glance which fields came from the request. Numbers and booleans pass through unquoted.

▮▮▮ CORS FIX ▮▮▮
  1. C-1

    The bug

    origin: process.env.CORS_ORIGIN || '*' with credentials: true. CORS_ORIGIN is never set by the deployed infra (verified with a repo-wide grep: only one reference, the consumer itself). So production was advertising "any origin allowed" — browsers ignore '*' when credentials are claimed, but it's the wrong default.

  2. C-2

    The fix

    Drop '*'. New DEFAULT_CORS_ORIGINS allowlist: localhost:5173 + localhost:3001 + their 127.0.0.1 twins (Vite dev + backend self-test) plus the production host (specodex.com apex and www). CORS_ORIGIN env override now parses as a comma-separated allowlist.

  3. C-3

    What's still permissive (and why it's not in scope)

    API Gateway in front of the Express Lambda still uses allowOrigins: ['*'] (app/infrastructure/lib/api-stack.ts:84). That's CDK — touching infrastructure under auto-mode is explicitly out of bounds per the orchestration rules. Tracked separately; not bundled here.
