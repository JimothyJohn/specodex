▮▮▮ WHAT CHANGES ▮▮▮
  1. B-1

    search.ts:55 — four interpolations, four strips

    q and manufacturer are zod z.string() (free-form); where and sort are z.union([z.string(), z.array(z.string())]) normalized to arrays. type is a zod-enum and limit is a number — both skip sanitization. Strip per element on the array cases before the join, so a CR/LF in any single entry doesn't survive.

  2. B-2

    products.ts:323 — dsUrl strip

    dsUrl is s3://<bucket>/<s3Key> where s3Key is built from the user-supplied filename in upload.ts. Inline strip before the delete-cascade log line.

  3. B-3

    dynamodb.ts:179 — SK strip on delete-failure

    item.SK?.S is string | undefined; the optional-chained replace + '<no-sk>' fallback covers both the missing-SK and the sanitized-SK cases. Reach is admin-only (deleteByScan is the helper for deleteByProductName) but the cost of the strip is one line, so defense-in-depth wins.

▮▮▮ WHY PREEMPT NOW ▮▮▮

These three sites have the same shape as alerts #60–#65 that PR #83 just closed; CodeQL's js/log-injection rule simply doesn't trace these particular flows on the current scan. That's not a guarantee — a future analyzer update or rule tweak could open them as new alerts on the next run. Fixing now also keeps the inline-replace pattern consistent across the backend, so a developer copying a nearby line doesn't accidentally re-introduce a bare console.log(`...${user}...`).
