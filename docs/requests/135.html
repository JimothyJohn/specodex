▮▮▮ PULL REQUEST #135 ▮▮▮

Removing a column now clears that column's filter.

Hiding a column via its × button used to leave any active filter on that attribute silently narrowing the row set — invisible from the UI but still living in filters state. Restoring the column later resurfaced the stale value. One-line fix to handleRemoveColumn in ProductList.tsx.

▮▮▮ WHAT WAS BROKEN ▮▮▮
  • Two pieces of state, one user action. The visible-column list (userHiddenKeys / userRestoredKeys) and the active-filter list (filters) are independent. Clicking × on a column header only touched the visibility set and the sort list — filters kept its entry for that attribute.
  • Silent narrowing. A user sets rated_voltage >= 200 V, hides the Rated Voltage column, and the row count stays narrowed. There's no chip or column to point at the filter; the product list just looks under-populated for no visible reason.
  • Stale value on restore. "+ Add Spec" brought the column back with the old filter value still applied — a value the user has no recent context for.
  • User-stated invariant. "Whenever a filter column is deleted its values should be reset to null so that the filters are no longer active." The previous behaviour violated the invariant; this PR restores it.
▮▮▮ THE FIX ▮▮▮

One line inside handleRemoveColumn in app/frontend/src/components/ProductList.tsx, paired with the existing sort/visibility cleanup: 

setFilters(prev => prev.filter(f => f.attribute !== attribute));

The handler now drops any FilterCriterion whose attribute matches the removed column. Restoring the column via "+ Add Spec" brings it back without a filter attached — same starting state as if the user had never touched that column.

Why a one-line fix is enough. handleRemoveColumn is the single entry point for the × button (called from ColumnHeader's onRemove at line 1233). Bulk-clear ("× All") flows through setFilters([]) already. There's no other surface where a column gets hidden without going through this handler.
