docs(build) — /build page spec, requirements-first system assembler

Adds todo/BUILD.md, a 1,710-line design spec for converting the /actuators page into /build — the second of three planned user-facing pages (Selection → Build → Wizard). Build is the requirements-first system assembler: the user enters engineering inputs (motion class, stroke, move time, dwell, payload mass), and the page narrows the catalogue per slot (actuator → motor → drive → optional gearhead) using a relations API to produce a copy-able BOM. Doc-only; no code, schema, or migrations touched.

▮▮▮ WHAT'S IN THE SPEC ▮▮▮
  1. P-1

    Part 1 — The three-page model

    Selection / Build / Wizard as altitudes of abstraction. Wizard (future) is the natural-language front door; Build (this spec) is the requirements-first system assembler; Selection (existing) is the raw catalogue browser / expert mode. Bidirectional funnel; BuildRequirements JSON is the shared currency between Wizard's output and Build's input.

  2. P-2

    Part 2 — Requirement vocabulary

    The load-bearing schema. BuildRequirements = motion class (linear / rotary), orientation (horizontal / vertical), payload mass, and a MotionProfile[] of stroke + move_time + dwell_time. Day 1 ships profiles[1]; the list shape is forward-compat for stacked moves later. Derivation engine assumes an S-curve 1/3 trapezoidal profile and computes peak velocity, peak acceleration, peak force, and duty cycle. URL-serialisable so Wizard can hand off via redirect.

  3. P-3

    Part 3 — Page surface

    Two-pane layout: <RequirementsForm> on the left (sticky), <CandidatesPane> on the right (per-slot accordion), <SystemSummary> as a sticky bottom strip absorbing today's BuildTray. Component prop boundaries pinned. The configurator drawer extracted from ActuatorPage surfaces only on matching Actuator candidate rows; stroke segment pre-populated from form context.

  4. P-4

    Part 4 — Relations API integration

    Four GET /api/v1/relations/* endpoints (actuators, motors-for-actuator, drives-for-motor, gearheads-for-motor) with wire shapes pinned. Frontend fan-out diagram. No client-side fallback — three reasons (volume, drift risk, API simplicity). Adds two items to SCHEMA Phase 3's scope: the new compatible_actuators() entry point and a _distribution_position block on candidate rows.

  5. P-5

    Part 5 — Selection's diminished role + the linear-actuator bug

    Three responsibilities move out of Selection (the rotary/linear/z-axis transmission buttons, the "Add to build" modal button, the BuildTray bottom strip). Documents the linear-actuator bug Nick reported (records exist in dev, exposed via categories endpoint, but absent from some user path) — most likely root cause is the UX trap with the conflated "Linear" transmission button. Diagnostic recipe pinned; fix is the strip-down itself.

  6. P-6

    Part 6 — Wizard handoff (forward context)

    Locks the seam, not Wizard's design. Wizard outputs a BuildRequirements URL; Build promises 4 numbered behaviours in return (pre-populate, auto-pick drafts, banner, drop the ?wizard=1 flag on user edit). Schema must stay backwards-compatible: new fields always optional, no field renames.

  7. P-7

    Part 7 — Migration & phasing

    Phase 1 = 4 PR sequence (1A relations API → 1B Selection strip-down → 1C Build MVP → 1D /actuators redirect). Phase 2 = vertical + gravity + friction + Save-to-Project + Build → Selection deep-link. Phase 3 = Wizard handoff scaffolding. 8-row retirement audit pinned (what files get deleted vs renamed). Three migration risks named: shared build slice, BUILD_SLOTS reorder, CDN SPA fallback.
