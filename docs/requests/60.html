▮▮▮ THE PERSISTED<T> TRANSFORM ▮▮▮

The generated TS has product_id?: string (optional, mirroring the Pydantic Field(default=None)). The API-facing union needs both product_id and product_type required. The transform distributes over the union and preserves discriminated narrowing: 

type Persisted<T> = T extends { product_type?: infer U }
    ? Omit<T, 'product_id' | 'product_type'> & {
            product_id: string;
            product_type: NonNullable<U>;
        }
    : never;

export type Product = Persisted<GeneratedProduct> | DatasheetEntry;
▮▮▮ THE FOUR MISMATCHES — RESOLVED ▮▮▮
  1. M-1

    product_id optional vs required

    Generated has ?: string (Pydantic optional). Hand-typed had it required. Persisted<T> tightens it for the API-facing union without changing the generated source.

  2. M-2

    product_name required vs optional

    The hand-typed ?: was always wrong — Pydantic Field(...) is required. Inherited from generated; test fixtures updated to include product_name.

  3. M-3

    ?: T | null vs ?: T

    Consumers already handled truthy-narrowing in most places. Two helper files (compat.ts) accept Maybe<T> = T | null | undefined so callers don't have to widen.

  4. M-4

    PK/SK in generated TS

    Resolved earlier in PR #58 — @computed_field dropped on Manufacturer/ProductBase/Datasheet; Pydantic now skips them in model_dump() and the JSON schema.
