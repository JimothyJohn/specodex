▮▮▮ PULL REQUEST #132 ▮▮▮

Unblock the deploy chain — backend Lambda bundle lockfile.

CI's Deploy Staging job has been failing on every master push since PR #130 was the first merge to actually run the deploy step to completion. Earlier merges all cancelled before reaching it. Result: prod hasn't shipped via CI since 2026-05-02 (stack timestamps confirm). The bug is in the Lambda-bundle step copying the workspace lockfile into a non-workspace context — fine until backend's jest bumped 29 → 30 (PR #19) and the divergent copy moved to a path npm couldn't reach.

▮▮▮ WHAT WAS BROKEN ▮▮▮
  • The deploy bundling step. cli/quickstart.py's cmd_deploy compiled app/backend with tsc into app/backend/dist/, then copied app/package-lock.json alongside backend/package.json and ran npm ci --omit=dev from inside dist/.
  • The workspace lockfile is workspace-shaped. When backend's jest bumped to 30.x while other workspaces kept 29.x, the lockfile carried the divergent copy under backend/node_modules/jest@30.3.0. From a non-workspace npm ci rooted at dist/, that path is unreachable — npm only sees node_modules/jest@29.7.0 at the top level.
  • npm bailed silently. The deploy step ran with --silent, so CI captured the exit code without the diagnostic. Reproducing locally surfaces it: Missing: @sinclair/typebox@0.34.49 from lock file, Missing: ci-info@4.4.0, Invalid: lock file's brace-expansion@1.1.12 does not satisfy 1.1.14, etc.
  • Why now. PR #130 was the first master push in days where CI actually ran the deploy step to completion (every prior push got cancelled by the next merge). The bug was latent since PR #19 merged jest@30.
▮▮▮ THE FIX ▮▮▮

Generate a dist/-rooted lockfile via npm install --package-lock-only immediately before the npm ci. This sidesteps the workspace-vs-non-workspace mismatch entirely — the lockfile in dist/ now describes the tree from backend/package.json's point of view, which is what npm ci in dist/ needs.

Trade-off: the previous comment valued the workspace lockfile's exact transitive pins as a guard against transient registry breakage (the @aws-sdk/xml-builder@3.972.20 + ESM-only @nodable/entities@2.1.0 incident on Node 18). We lose that here — backend's prod transitives now resolve fresh from the registry on each deploy. Acceptable because Node 20 supports ESM-in-require, dependabot already churns these pins weekly, and a future incident can be pinned surgically via overrides in backend/package.json.

  • Local repro of the failure: from app/backend, npm run build → cp package.json dist/ → cp ../package-lock.json dist/ → cd dist && npm ci --omit=dev --ignore-scripts --no-audit --no-fund. Bombs with Missing: @sinclair/typebox@0.34.49 from lock file.

  • Regression sweep: ./Quickstart verify --only python still passes (1435 tests) — the change is in the deploy code path which the verify suite doesn't currently exercise. Follow-up worth filing: have verify exercise the bundle step too, so a regression here doesn't sit latent for days again.
