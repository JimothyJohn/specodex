▮▮▮ THE TWO BUGS ▮▮▮
  1. B-1

    Real exponential-backoff waits were running

    The @retry decorator on generate_content uses wait_exponential(min=4, max=60). Multiplied by 5 attempts, that's 20-300s of real waiting per failing call. test_invalid_content_type_raises had the same problem inadvertently — it raises ValueError inside the retry decorator, and tenacity retries every exception by default.

  2. B-2

    Cross-test lru_cache pollution

    specodex.llm._client_for is @lru_cache(maxsize=4). When a prior test populated the cache with "fake-key", this test's @patch("specodex.llm.genai") didn't intercept the call — the cache returned a stale Client and the real google.genai built a UserContent that failed on empty parts.

▮▮▮ THE FIX ▮▮▮

Module-scoped autouse fixture _no_retry_waits that:

  • Clears the lru_cache before each test in the module so @patch("specodex.llm.genai") is the source of truth.
  • Mutates generate_content.retry.wait → wait_none() so tenacity retries fire instantaneously.
  • Restores both in finally so the patch doesn't leak into the rest of the suite.

Module-scope (not per-test) because two tests in the file exercise the retry path and a third inadvertently does too — keeping the override per-file scopes the patch correctly without affecting other tests that legitimately want backoff timing.
