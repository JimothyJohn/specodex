# DB schema cleanup ‚Äî gameplan

Driven by the field-coverage audit at
`outputs/audit_fields_dev_<ts>.md` (re-runnable via
`uv run python -m cli.audit_fields --stage dev`). The audit scans
`products-dev` and tabulates, per `product_type`, the fraction of rows
that have a populated value for every Pydantic-declared field.

**Status (post-2026-05-10 sprint):** Phase 1 ‚úÖ shipped (gearhead
torque rename + electric_cylinder field drops). **Phase 2 needs a
decision before implementation** ‚Äî the audit memory
(`project_field_audit_findings.md`) recommends *dropping* `lead_time` /
`warranty` / `msrp` outright (0% populate rate, fields inherited
speculatively from a Stripe-driven design that never landed). The
churn plan's previous framing of "Phase 2 = populate these fields" is
stale. Decide drop-vs-populate before opening a PR.

This doc enumerates the cleanup phases that fell out of the
2026-05-07 audit (against 6578 PRODUCT rows, 6 product types) and which
of them this PR ships.

## What's in scope for this PR

### Phase 1 ‚Äî Gearhead torque field-name fix (real bug, ships)

The `gearhead` model declares `rated_torque` (0% hits) and
`peak_torque` (0.6% hits). The Gemini extractor emits
`max_continuous_torque` (51.8%) and `max_peak_torque` (51.2%) ‚Äî and
those land in DynamoDB on PRODUCT# rows but the Pydantic model can't
read them back. The API and frontend never see the data even though
51%+ of gearhead rows have it.

The `specodex/models/gearhead.md` ADR (lines 41‚Äì42, 87, 107) already
documents `max_continuous_torque` / `max_peak_torque` as the canonical
names, so the bug is squarely in `gearhead.py` having drifted from its
own ADR.

**Files touched:**

- `specodex/models/gearhead.py` ‚Äî rename the two fields
- `specodex/integration/adapters.py` (line 106‚Äì107) ‚Äî gearhead adapter
  maps `Gearhead.rated_torque` ‚Üí port's `rated_torque`. Keep the port
  field name (`MechanicalShaftPort` is shared with motor and motor
  rightly uses `rated_torque`); only update the source side
- `specodex/spec_rules.py` (lines 53‚Äì54) ‚Äî add bounds entries for
  `max_continuous_torque` and `max_peak_torque`. Leave the existing
  `rated_torque` / `peak_torque` rules in place (still correct for
  motor)
- `cli/intake.py` (line ~57) ‚Äî gearhead column-list comment
- `app/frontend/src/types/generated.ts` ‚Äî regenerated by
  `./Quickstart gen-types`

**No DB migration needed.** Existing rows already carry the new field
names; the rename just lets the API see them. The 0‚Äì0.6% rows under
`rated_torque`/`peak_torque` are noise (likely from the
sub-1% intake before this drift was ever caught) ‚Äî they'll persist
harmlessly until next dedupe pass.

### Phase 2 ‚Äî Drop two semantically-wrong fields from electric_cylinder (ships)

The `electric_cylinder` model declares two fields that don't apply to
the product class and have 0% hit rate after 265 rows:

- `gear_ratio` ‚Äî electric cylinders don't expose a gear ratio in the
  rotary sense (lead-screw pitch is the relevant linear analogue, and
  that field already exists). Likely inherited speculatively.
- `linear_speed_at_rated_load` ‚Äî duplicates `max_linear_speed` for
  practical purposes; the LLM never picks up a separate value.

**Files touched:**

- `specodex/models/electric_cylinder.py` ‚Äî drop both fields
- `app/frontend/src/types/generated.ts` ‚Äî regenerated

### Phase 3 ‚Äî Field-coverage audit CLI (ships)

`cli/audit_fields.py` is the audit tool that surfaced these. It's
read-only (scan + report only, no DB writes), filters to PK
`PRODUCT#*` to avoid double-counting `INGEST#*` co-tenant rows, and
mirrors the `audit_dedupes` CLI shape so it's re-runnable from the
same console pattern. Output goes to
`outputs/audit_fields_<stage>_<ts>.{json,md}`.

## What's deferred (called out, not shipped)

### `lead_time` and `warranty` on `ProductBase` ‚Äî keep

Both are 0‚Äì13% across product types. Datasheets don't carry this
data; it would come from a distributor / catalog enrichment path that
isn't running at scale. **Per Nick's instruction, leave them in.**
They're cheap to keep (Optional fields with no extractor cost) and
the path to populating them is a separate workstream, not a model
cleanup.

### `msrp` / `msrp_source_url` / `msrp_fetched_at` on `ProductBase` ‚Äî keep

`./Quickstart price-enrich` exists and does populate these (drive
shows 0.7% on `msrp` ‚Äî a small but real signal that enrichment is
sometimes running). Removing them would tear out the price-enrich
contract. Keep until / unless that workstream is explicitly retired.

### `linear_actuator` over-spec ‚Äî defer

13/43 declared fields at 0%. Sample is only 46 PRODUCT# rows so the
threshold-based call is shaky; before dropping fields like
`holding_force`, `peak_current`, `rated_voltage`, `dynamic_load_rating`
we should ingest a few more vendors and re-run the audit. Likely
candidates for a future cleanup pass once n is meaningful.

### `drive` borderline fields ‚Äî keep

`ethernet_ports` (2.1%) and `release_year` (2.5%) are below the 5%
threshold but informative when present (PLC-class drives, model
recency). No win in dropping.

### `robot_arm` per-type fields ‚Äî keep

`force_torque_sensor` (2.0%), `dimensions` (2.2%), `product_family`
(0%) are all structurally sparse rather than wrong. Robot arms are
typically family-named (so `product_family` empty is normal ‚Äî the
`product_name` carries the family), only some have integrated F/T
sensors, and very few datasheets express bounding-box dimensions for
6-DOF arms. Keep.
